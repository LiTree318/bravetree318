#### 摘要


儘管 Obsidian 有助於捕捉細節，但==不要過度迷失在細節裡面==。外掛或者佈景主題可以研究，但這應該只是==輔助和優化==；[[資料庫動態框架路徑：主題、發展、精鍊與進階\|而非拘泥打造出完美的 Vault]][^3]。

其次，[[Data Frame.png|我規劃以 input 為主體的 process]][^1]，發現從 **Library** 到 **Output** 之間會卡住；因為蒐集過多的 **Input** 筆記導致整理的過程中失能。

因此，我需要另外研究[[化繁為簡與化簡為繁：比較 input 導向與 output 導向的資料庫框架\|以 output 為主體的 process]][^2]。

## 前言：Obsidian 的優勢與劣勢




Obsidian 的優勢在於能透過 **internal link** 快速設立不同筆記之間的關聯性，這個特點使得 Obsidian 在處理資料時能更適應破碎、零散的資訊來源。特別是當我們在短時間內漫無目的地蒐集資料、發想點子時，不同資訊和靈感之間尚未能構成足夠完整的篇幅，這時候，透過 internal link 功能，可以快速將這些分散的點子串連起來。


> ![Pasted image 20260210234109.png](/img/user/Pasted%20image%2020260210234109.png)
> 
> 圖：[[筆記軟體方法論/使用 Obsidian 做筆記的資訊超載問題\|使用 Obsidian 做筆記的資訊超載問題]] 中串連的其他概念和文件。


開啟 interna link 功能的方式可以透過 `cmd + P` 或者從左側選單選取 `Open command palette`；不過我建議設定快捷鍵，點選選單打開 `hotkeys` ，從搜尋欄中尋找 `Add internal link`，來設定快捷鍵（我這邊是設定 `cmd + alt + K`）。

設定快捷鍵的優點在於，我們可以在書寫的過程中，隨時延伸新的想法和點子。例如在[[筆記軟體方法論/使用 Obsidian 做筆記的資訊超載問題#摘要\|本文開頭摘摘要裡]]，[[簡報化的文字、便利貼式的筆記\|我就是在撰寫文章摘要的過程中，延伸了三個想法，也預計是接下來本文完成之後的三篇文章]][^4]。
更進一步，結合 `Graph View` 可以清楚看見當前文件與其他筆記之間的**關聯性**，有助於我們能快速在繁雜的資料庫中找到相關聯的筆記。

然而，隨著整理越來越多資料，資料庫（或者電腦資料夾）會越來越混亂；這在 Obsidian 這種以零碎、分散式為特色的筆記軟體中問題會更加嚴重。這會降低筆記（以及筆記軟體）的效用，同時也會製造大量無效筆記，降低我們從資料庫中尋找有效資訊的效率。


> ![Pasted image 20260211000300.png](/img/user/Pasted%20image%2020260211000300.png)
> 圖：過多資料、文件使得資料庫膨脹到難以處理。
> 其中：淺灰色節點是筆記、深灰色節點是相關聯但未新增的筆記、黃色節點是附件和檔案、綠色節點則是標籤（tags）。


為了解決這個問題，我們需要在新增筆記的過程中引入系統化的管理流程，讓資料庫中的每一則筆記都能有明確的定位和用途。而筆記管理流程的基礎會影響我們如何看待筆記的性質，以及每則筆記在資料庫中的具體作用；**換言之，訂定筆記管理流程也就意謂著我們將界定這個資料庫的核心及方法論**。

## 以 Input 為導向的方法論




Obsidian 有利於我們整理繁雜的資料來源[^5]和靈感，當這些內容以筆記的形式導入資料庫裡面時，他們會是資料來源；亦即，這些資料和靈感是我們後續在發想作品或者創作時，重要的參考資料。因此，建立以資料來源等「輸入內容」（input）為核心的資料庫方法論，能更快速整理資料庫的框架和內容。

在建立以 input 為導向的筆記管理流程之前，我們需要先確認 input和筆記之間的關聯性。我們可以參照下圖：「以 Input 為導向的資料庫流程與架構」，從圖所示，當我們將資料來源導入資料庫時，它會以**引文**（quotes）、**擷取**（clips）**和片段**（shorts）等形式變成筆記的內容；而當我們詮釋這些資料（如：新增心得、想法，或者評論），結合原始資料和詮釋內容的筆記是**註解**（citation）。這些經過我們憑註的筆記會構成我們資料庫的基礎，理想上，資料庫中的每一則筆記都有著對應資料來源以及我們的評論。

而當我們將這些筆記的內容應用於創作中時，可以透過在創作中以 internal link 的方式新增反向連結，連結**註解**與創作內容；而一份包含各式各樣註解筆記的成果，便是**產出**（output）。

承接上述，我們可以建立一條輸入到產出的筆記歷程，其中，每個階段都標誌著不同意涵的筆記：
1. 資料階段一，筆記內容為擷取資料來源、網頁、文件等。
2. 資料階段二，筆記內容是從前階段（第一階段）擷取出的內容（引用段落、擷取片段等等）
3. 資料階段三，筆記內容會包含第二階段擷取的段落，以及我們撰寫的評論或者註解。
4. 資料階段四，成果或者草稿。


> ![Data Frame.png](/img/user/ignore/img/Data%20Frame.png)
> 圖、以 Input 為導向的資料庫流程與架構。



由輸入（input）到輸出（output），資料階段越前面越接近輸入來源，越後面則越接近產出的成果。以此為基礎，我們可以建立一套筆記的「資料生命歷程」（Life Course of Data/Information，LCD/I）。精簡上述四個階段，可以得到三個不同階段的 LCD/I：
1. $T_1$：該筆記包含原始資料和擷取片段（對應資料階段一、二）；
2. $T_2$：該筆記包含摘要內容和評論（對應資料階段二、三）；
3. $T_3$：該筆記包含複數的摘要內容（例如一篇 critical review），或者是階段性的草稿（對應資料階段三、四）。


此時，我們可以為每一則筆記設立 `Metadata` 來區分不同筆記的主題和資料階段。透過快捷鍵 `cmd + ;`（或者從指令中找到 `Add file property`），新增每則筆記的 `metadata`。每則筆記透過 `tags` 來標註不同筆記的主題；以 `LCDI_step` 來標註該篇筆記對應的資料片段。


> [!example] 參考用的筆記 metadata，
> ==請放在每則筆記的第一行==
> 
> ```markdown
> tags:
>   - obsidian-method
> LCDI_step:
>   - "01 Ref. or Quote"
>   - "02 Citation"
>   - "03 Review or Draft"
> ```
> 
> `tags` 是筆記的主題，以列表（list）的方式新增筆記關連的主題或者關鍵字。
> 
> 而 `LCDI_step` 則是筆記對應的資料歷程，每份筆記會對應 1 ~ 3 其中一個階段。

## 治絲益棼：資料庫 basement 以及其適用性問題


> [!warning] 
> 1. 增加單一筆記的複雜度，提高使用筆記軟體的技術門檻。
> 2. 著重在定位（position），反而抵消以零碎見長的軟體特點。
> 3. LCD/I 體系更強調的是 input 本身在database中的角色，換言之，是以 **資料來源處理到什麼進度** 作為資料庫的底層架構。當 database 服務於資料來源時，筆記軟體會成為失去筆記的功能，而只剩下類似 RSS 的功用──彙整資料而無法處理。
> 4. 核心課題在於如何==將整理好的資料應用於創作作品中==，以 input 為導向的體系缺少對於 output 的規劃；這將使得資料庫存在大量冗餘、無用的資料和[[為標籤管理系統撰寫說明文件之必要性\|標籤]]。
> 5. 因此，LCD/I 體系僅只解決了 input 的問題，但卻會因為需要管理 input 而創造大量的標籤，和無用、無意義的資料節點（例如：摘錄一句話，並撰寫心得、評論的筆記）。
> 6. 臃腫的資料庫會難以整理，最終無法使用：亦即整個資料庫會因為過度細碎的資料而超載、當機。
>

## 小結：資料庫框架與資料主題的類型

> [!tip] 
> 1. 檢討資料庫框架：關乎什麼資料或主題的資料庫？
> 2. 資料和主題的差異性：發散式、聚焦式、其他
> 3. 資料或主題的形態差異如何影響資料庫框架
> 4. 動態調整與交互作用：資料主題類型、資料庫框架與資料庫
> 5. 小結：input 的貢獻（拆解引述和轉譯資料的流程，方便檢索與追蹤），與侷限（無法刺激產出，資訊超載）。
> 6. 解方與侷限：以刺激產出為主體建構資料庫框架，回頭收束資料與資料庫。
>
